# База данных кредитного процесса

## Общее описание
Данная база данных предназначена для автоматизации кредитного конвейера: от заявки до обслуживания договоров, управления просрочками, реструктуризациями, резервами, документооборотом и задачами сотрудников. Реализована на **PostgreSQL 18** в схеме `public2` с использованием пользовательских типов (ENUM) для строгой типизации статусов и категорий.

## Основные сущности и связи

### 1. Заёмщики (`borrowers`)
Хранит персональные данные клиентов: ФИО, телефон, паспорт, email, кредитный скоринг, доход, статус (`активный`, `проблемный`, `архив`), дату регистрации.  
**Ограничения**: уникальные email, паспорт.

### 2. Кредитные продукты (`credit_products`)
Типы кредитов: название, тип (`ипотека`, `потребительский`, `авто`, `pos`), минимальная/максимальная сумма, ставка, сроки, признак активности.

### 3. Лимиты (`limits`)
Лимиты кредитования: тип (`региональный`, `отраслевой`, `продуктовый`), значение, валюта, период (`день`, `месяц`, `квартал`), срок действия, активность.  
Связь с продуктами через `limit_products` (многие-ко-многим).

### 4. Сотрудники (`employees`)
Учётные записи сотрудников: логин, ФИО, роль (`аналитик`, `риск-менеджер`, `коллекшен`, `руководитель`), департамент, email, активность.

### 5. Кредитные заявки (`credit_applications`)
Заявки клиентов: заёмщик, продукт, лимит, ответственный сотрудник, сумма, срок, цель, статус (`новая`, `рассмотрение`, `одобрена`, `отклонена`), комментарий, дата подачи.

### 6. Кредитные договоры (`credit_agreements`)
Заключённые договоры: заявка, заёмщик, номер, сумма, ставка, срок, даты начала и окончания, статус (`активен`, `просрочка`, `закрыт`, `реструктуризация`).

### 7. Платежи (`payments`)
График платежей по договору: плановая дата, фактическая дата, сумма, тип (`основной`, `проценты`, `штраф`), статус (`ожидается`, `оплачен`, `просрочен`).

### 8. Просрочки (`delays`)
Информация о просрочках: договор, ответственный менеджер, сумма, количество дней, стадия (`1-30`, `31-60`, `61-90`, `90+`), причина, дата возникновения, статус (`новая`, `в_работе`, `в_суде`, `урегулирована`).

### 9. Реструктуризации (`restructurings`)
Данные о реструктуризации договора: тип (`отсрочка`, `уменьшение_платежа`, `пролонгация`), старая/новая ставка, новый срок, дата, статус (`заявка`, `одобрена`, `отклонена`).

### 10. Резервы (`reserves`)
Резервы по договорам: сумма, тип (`индивидуальный`, `портфельный`), метод расчёта, дата расчёта, статус (`сформирован`, `скорректирован`, `списан`).

### 11. Документы (`documents`)
Файлы документов: автор, название, тип (`паспорт`, `справка`, `договор`, `отчёт`), путь, дата загрузки.  
Связи с заёмщиками, заявками и договорами через таблицы `document_borrowers`, `document_applications`, `document_agreements` соответственно.  
Права доступа к документам регулируются через `document_access` (сотрудник, уровень: `чтение`, `запись`, `удаление`).

### 12. Задачи (`tasks`)
Рабочие задачи: исполнитель, создатель, описание, дедлайн, приоритет (`низкий`, `средний`, `высокий`, `критичный`), статус (`новая`, `в_работе`, `завершена`, `просрочена`).  
Связи с заявками, договорами и просрочками через таблицы `task_applications`, `task_agreements`, `task_delays`.  
Комментарии к задачам хранятся в `task_comments`.

## Ключевые особенности
- **ENUM-типы** для всех статусов и категорий обеспечивают целостность данных и удобство использования.
- **Уникальные ограничения** на ключевые поля (email, паспорт, логин, номер договора).
- **Внешние ключи** с каскадными или ограничивающими действиями для поддержания ссылочной целостности.
- **Связующие таблицы** для документов и задач реализуют гибкие связи многие-ко-многим.
- **Аудиторские поля** `created_at` фиксируют время создания записей.
